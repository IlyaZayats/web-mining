Тип 3

Проект: Backend-проект по Go (уровень Junior+)

Producer → Kafka → Consumer → БД + HTTP API + кэш. Логирование, тестирование, конфигурация и graceful shutdown.

К структуре

Цель проекта

Разработать backend-систему на Go, применив изученные темы: HTTP API, работа с БД, работа с сообщениями (Kafka), логирование, тестирование, кэширование и корректное завершение приложения.

ТЗ (описание задачи)

Необходимо реализовать систему из двух компонентов:

  1. Producer-сервис, публикующий события в Kafka (например: создание/обновление сущности).
  2. Основной сервис (API + Consumer), который читает события из Kafka, сохраняет данные в БД, отдаёт данные через HTTP API и использует кэш для ускорения чтения.
Система должна запускаться локально. Kubernetes не используется. Docker — по желанию (docker-compose приветствуется).
Функциональные требования
  • Producer публикует события в Kafka (создание/обновление сущности).
  • Consumer читает события и сохраняет их в БД (минимум: INSERT/UPDATE).
  • HTTP API предоставляет CRUD или Read-heavy API для одной доменной сущности (Tasks/Products/Orders).
  • GET-эндпоинты читают данные из БД и используют кэш (cache-aside) с TTL.
  • Единый формат ошибок (JSON) и корректные HTTP статусы.
  • Валидация входных данных (минимум для POST/PUT).
  • Логирование: входящие запросы, обработка сообщений, ошибки БД/Kafka.
Нефункциональные требования
  • Конфигурация через env-файл или переменные окружения (DSN БД, адрес Kafka, порты, TTL кэша).
  • Graceful shutdown: завершение HTTP-сервера и Consumer без потери данных (в разумных пределах).
  • Документация API: OpenAPI/Swagger или подробный README с примерами запросов.
  • Минимальная безопасность: API key или JWT для изменяющих операций (POST/PUT/DELETE).
  • Идемпотентная обработка сообщений (желательно).
  • Надёжность: таймауты на запросы к БД/Kafka/кэшу, обработка временных ошибок.

Требования

  • Язык: Go 1.21+ (рекомендуется 1.22+), Go modules.
  • HTTP: net/http или роутер (chi/gin/echo) — на выбор.
  • Kafka: Apache Kafka или совместимый брокер (например, Redpanda) для локального запуска.
  • Kafka client: segmentio/kafka-go или confluent-kafka-go — на выбор.
  • База данных: PostgreSQL (рекомендуется) или другая реляционная БД по согласованию.
  • Доступ к БД: database/sql (+ pgx) или ORM (например, GORM) — на выбор.
  • Кэш: Redis (желательно) или in-memory кэш (LRU/TTL); обязательно TTL и cache-aside.
  • Логи: log/slog или структурный логгер (zap/zerolog) — на выбор.
  • Docker: опционально (docker-compose приветствуется для Kafka/DB/Redis).

  • Unit-тесты для бизнес-логики (минимум: сервисный слой).
  • Тесты репозитория/хранилища: mock или integration тесты на реальной БД.
  • Тесты HTTP-обработчиков через httptest: статусы, формат ответов и ошибок.
  • Тестирование consumer-логики: минимум 1 тест обработки сообщения (можно с mock Kafka).
  • Команда go test ./... должна проходить без ошибок.

  • Ссылка на репозиторий (GitHub/GitLab/другое) с исходным кодом.
  • Отчёт (PDF/DOCX/MD): архитектура, схема данных, описание сообщений Kafka, описание кэша, инструкция запуска.
  • README: запуск локально (и с Docker при наличии), примеры запросов (curl/HTTPie), описание конфигурации.
  • Опционально: скриншоты/примеры запросов и ответы сервиса.

Пример доменной сущности

Tasks
id, title, status, created_at
Products
id, name, price, updated_at
Orders
id, customer_id, total, status

Пример событий Kafka

Формат сообщений — JSON
{"event":"task_created","id":123,"title":"Write docs","status":"new","ts":"2026-02-05T12:00:00Z"}

Критерии оценивания

Критерий Вес
Корректность HTTP API (контракты, статусы, формат ошибок, валидация) 30%
Работа с Kafka (producer + consumer, обработка ошибок, корректная запись в БД) 25%
Работа с БД (схема, миграции, транзакции/конкурентность при необходимости) 20%
Кэширование (стратегия cache-aside, TTL, актуализация/инвалидация по событиям) 15%
Тесты и документация (README/отчёт, go test, примеры запросов) 10%
Условия зачёта (минимум)
  • Сервис запускается локально.
  • Producer публикует события.
  • Consumer сохраняет данные в БД.
  • HTTP API отдаёт данные из БД; чтение использует кэш.
  • Есть тесты и документация (README и отчёт).
Штрафы: отсутствие инструкции запуска, нерабочие тесты, плагиат.
